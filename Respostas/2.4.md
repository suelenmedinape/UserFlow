
### Importância do `trackBy`

Por padrão, quando a lista de itens é reatribuída (ex: chegou uma nova resposta da API), o Angular não sabe o que mudou e o que não mudou. Ele joga todos os elementos do DOM fora e recria do zero, mesmo que 99% dos itens sejam idênticos.
O trackBy é uma função que diz ao Angular: "Use este identificador único para reconhecer cada item. Se o id for o mesmo, o elemento já existe no DOM, não recrie."

```typescript
// No componente
trackByPessoaId(index: number, pessoa: Pessoa): number {
 return pessoa.id;
}

<!-- No template -->
@for (pessoa of pessoas; track pessoa.id) {
 <app-pessoa-card [pessoa]="pessoa" />
}
<!-- Sintaxe legada com trackBy -->
<li *ngFor="let pessoa of pessoas; trackBy: trackByPessoaId">
```

### `ChangeDetectionStrategy.OnPush` pode reduzir ciclos desnecessários

Se o usuário clica em um botão no componente PAI e nenhum @Input() dos filhos mudou, o Angular pula todos os 500 PessoaCardComponent na verificação. O ganho de performance é grande.

```typescript
import { Component, Input, ChangeDetectionStrategy } from '@angular/core';

@Component({
 selector: 'app-pessoa-card',
 template: `<p>{{ pessoa.nome }}</p>`,
 changeDetection: ChangeDetectionStrategy.OnPush
})
export class PessoaCardComponent {
 @Input() pessoa!: Pessoa;
}

```

### O impacto de usar a estratégia Default

Com ChangeDetectionStrategy.Default, o Angular usa uma abordagem pessimista: a cada evento disparado, seja um clique, uma resposta HTTP ou até um setInterval, ele percorre e verifica todos os componentes da árvore, do pai ao filho mais profundo.
Em uma lista de 500 itens, isso significa que, quando o usuário digita no campo de busca, o Angular verifica os 500 componentes filhos, mesmo que nenhum @Input() deles tenha mudado.
